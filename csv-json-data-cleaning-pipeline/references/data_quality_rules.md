# 数据质量规则说明

## 1. 文档目的

本文档用于说明 CSV/JSON数据批量清洗产线中可采用的基础数据质量规则，帮助数据开发人员在配置清洗任务时保持统一口径。

本 Skill 关注中级数据开发项目中常见的数据质量问题，不涉及复杂模型训练、机器学习预测修复或高级语义审核。

## 2. 质量规则分类

### 2.1 完整性规则

完整性规则用于检查数据是否缺少关键字段或关键字段值。

常见检查项：

- 必填字段是否存在。
- 必填字段是否为空。
- JSON 或 JSONL 记录是否字段结构不一致。
- CSV 表头是否缺失。

处理建议：

- 关键字段缺失时，优先输出到 `issue_rows`。
- 可补充字段缺失时，可使用默认值填充。
- 无法判断的数据应保留问题原因，交由人工复核。

### 2.2 唯一性规则

唯一性规则用于识别重复数据。

常见检查项：

- 单主键重复，例如 `id` 重复。
- 组合字段重复，例如 `title + source` 重复。
- 同一批次内重复。
- 与历史数据重复。

处理建议：

- 明确保留策略，例如保留第一条、保留最后一条或保留更新时间最新记录。
- 被去除的重复记录应输出到 `issue_rows`。
- 去重规则应写入 `cleaning_log`，便于追溯。

### 2.3 有效性规则

有效性规则用于检查字段值是否符合业务范围或格式要求。

常见检查项：

- 日期是否能被解析。
- 金额是否为合法数值。
- 手机号、邮箱、编码是否符合格式。
- 数值是否超出合理范围。
- 字符串长度是否过短或过长。

处理建议：

- 可明确修复的数据进行标准化处理。
- 无法修复的数据输出到 `issue_rows`。
- 对范围类异常应记录阈值和原始值。

### 2.4 一致性规则

一致性规则用于检查字段之间或不同来源之间是否存在冲突。

常见检查项：

- 开始时间是否晚于结束时间。
- 状态字段和日期字段是否矛盾。
- 分类字段和标签字段是否不匹配。
- 同一记录的多个来源字段是否冲突。

处理建议：

- 轻微不一致可标记后保留。
- 影响业务使用的不一致数据应输出问题清单。
- 规则说明中应写清楚判断依据。

### 2.5 标准性规则

标准性规则用于统一字段格式、字段名称和字段取值。

常见检查项：

- 日期格式是否统一。
- 金额是否统一小数位。
- 枚举值是否符合标准字典。
- 字段名称是否符合目标表结构。
- 空值表示是否统一。

处理建议：

- 优先使用字段映射和数据字典进行统一。
- 原始值和标准值应在问题记录或日志中保留。
- 不确定的映射不应自动替换，应进入人工复核。

### 2.6 可追溯性规则

可追溯性规则用于保证清洗过程可回看、可解释。

常见检查项：

- 是否记录清洗任务名称。
- 是否记录输入文件和输出文件。
- 是否记录执行规则。
- 是否记录影响数据量。
- 是否记录问题原因和处理动作。

处理建议：

- 每个清洗步骤都应写入 `cleaning_log`。
- 问题数据应保留原始行号或记录标识。
- 清洗统计应能反映清洗前后的数据量变化。

## 3. 问题类型建议

建议在 `issue_rows` 中使用统一的问题类型，便于统计和后续治理。

| issue_type | 含义 | 示例 |
|---|---|---|
| missing_field | 缺少字段 | 缺少 `title` 字段 |
| missing_value | 字段值缺失 | `content` 为空 |
| duplicate | 重复数据 | `id` 重复 |
| invalid_format | 格式错误 | 日期无法解析 |
| enum_invalid | 枚举值非法 | `status=完成中` 不在字典内 |
| out_of_range | 数值越界 | `score=120` |
| inconsistent | 字段逻辑不一致 | 开始时间晚于结束时间 |
| low_quality | 低质量内容 | 正文过短或无意义 |

## 4. 处理动作建议

| action | 含义 | 适用场景 |
|---|---|---|
| mark | 标记问题但保留数据 | 需要人工复核的数据 |
| drop | 删除数据 | 关键字段缺失或无效数据 |
| fill | 填充默认值 | 来源、状态等可补充字段 |
| standardize | 标准化处理 | 日期、金额、枚举值 |
| export | 导出问题数据 | 重复、异常、无法修复数据 |
| ignore | 忽略处理 | 非关键字段或业务允许为空 |

## 5. 规则配置原则

1. 先处理字段结构，再处理字段值。
2. 先做精确规则，再做模糊或人工判断规则。
3. 对关键字段采用更严格的处理策略。
4. 对无法确定的数据不要强行修复，应输出问题清单。
5. 每一类规则都应能在统计报告中体现。
6. 所有删除、填充、标准化动作都应写入日志。

## 6. 与标准要求的对应关系

| 标准要求 | 本 Skill 对应能力 |
|---|---|
| 数据去重 | `unique_keys` 和重复数据检查 |
| 缺失处理 | `required_fields` 和 `null_handling` |
| 内容过滤 | `anomaly_rules` 和低质量内容规则 |
| 格式标准化 | `date_format`、`phone_format`、`amount_format`、`enum_rules` |
| 清洗追溯 | `cleaning_log`、`cleaning_summary`、`issue_rows` |
